---
title: "Error Handling"
description: "Understand and handle errors returned by ElastomerClient"
---

ElastomerClient classifies every error into one of two categories: **fatal** errors that indicate a fundamentally broken request, and **retryable** errors that may succeed if tried again. All errors inherit from `ElastomerClient::Client::Error`.

ElastomerClient makes **no attempt to retry operations automatically**. Retry logic ‚Äî including backoff strategy and retry limits ‚Äî is left entirely to the caller.

---

## The base error class

All client errors inherit from `ElastomerClient::Client::Error`, which itself inherits from `ElastomerClient::Error` (a subclass of `StandardError`).

```ruby
begin
  client.docs("my-index").search({ query: { match_all: {} } })
rescue ElastomerClient::Client::Error => err
  puts err.message   # Human-readable error message
  puts err.status    # HTTP status code (Integer or nil)
  puts err.error     # Elasticsearch error body (Hash or nil)
  puts err.fatal?    # true if request should not be retried
  puts err.retry?    # true if request may be retried (inverse of fatal?)
end
```

### `error.status`

The HTTP status code from the Elasticsearch response, e.g. `404`, `503`. Returns `nil` if the error was not produced from an HTTP response (e.g. a connection failure).

### `error.error`

The `"error"` field from the Elasticsearch response body as a Hash, or `nil`. This contains structured details such as the Elasticsearch exception type, reason, and root cause.

```ruby
rescue ElastomerClient::Client::Error => err
  if err.error
    type   = err.error.dig("root_cause", 0, "type")
    reason = err.error.dig("root_cause", 0, "reason")
    puts "#{type}: #{reason}"
  end
end
```

### `error.fatal?` and `error.retry?`

```ruby
rescue ElastomerClient::Client::Error => err
  if err.fatal?
    # The request is fundamentally wrong ‚Äî fix the request before retrying
    raise
  else
    # Transient failure ‚Äî safe to retry with backoff
    sleep(1)
    retry
  end
end
```

<Warning>
  Most errors are **fatal by default**. Only `ServerError`, `TimeoutError`, `ConnectionFailed`, and `RejectedExecutionError` have `fatal?` set to `false`. Blindly retrying a fatal error will not help and may make things worse. Always check `err.fatal?` before retrying.
</Warning>

## Implementing retry logic

Because ElastomerClient does not auto-retry, you must implement your own strategy. A minimal retry loop:

```ruby
begin
  docs.search(query)
rescue ElastomerClient::Client::Error => err
  raise if err.fatal?
  retry # implement your own backoff
end
```

A more complete example with exponential backoff:

```ruby
MAX_RETRIES = 3

retries = 0
begin
  docs.search(query)
rescue ElastomerClient::Client::Error => err
  raise if err.fatal?
  raise if retries >= MAX_RETRIES

  retries += 1
  sleep(2**retries)  # 2s, 4s, 8s
  retry
end
```

<Note>
  Retrying during cluster overload adds more load to a cluster that is already struggling. Consider implementing circuit-breaker logic or back-pressure signalling rather than unconditional retries.
</Note>

## Handling specific error types

```ruby
begin
  client.docs("orders").get(id: "123")

rescue ElastomerClient::Client::IndexNotFoundError
  # The index doesn't exist yet ‚Äî create it or handle gracefully
  puts "Index not found"

rescue ElastomerClient::Client::TimeoutError
  # Network timeout ‚Äî safe to retry
  puts "Request timed out, retrying..."
  retry

rescue ElastomerClient::Client::RejectedExecutionError
  # Cluster thread pool is full ‚Äî back off and retry
  sleep 5
  retry

rescue ElastomerClient::Client::ServerError => err
  # 5xx from Elasticsearch ‚Äî may be transient
  puts "Server error #{err.status}, will retry"
  retry

rescue ElastomerClient::Client::Error => err
  # Catch-all for any other client error
  raise if err.fatal?
  retry
end
```

## OpaqueId header mismatch

When you initialize the client with `:opaque_id => true`, ElastomerClient sets a unique `X-Opaque-Id` header on every request and validates that the same value is echoed back in the response. If the headers disagree ‚Äî which can happen with persistent connections that return stale data from a previous request ‚Äî ElastomerClient raises an error and **automatically resets the connection** to prevent data leaks.

```ruby
client = ElastomerClient::Client.new(
  url:       "http://localhost:9200",
  adapter:   :net_http_persistent,
  opaque_id: true
)
```

If the `X-Opaque-Id` headers do not match, the middleware raises an `OpaqueIdError` and drops the persistent connection so that the next request starts fresh. You do not need to handle this error specially ‚Äî the connection reset happens automatically.

<Info>
  You only need `:opaque_id` when using persistent HTTP adapters (e.g. `:net_http_persistent`). With standard non-persistent connections each request uses a new socket and stale response data is not a concern.
</Info>

## Faraday error wrapping

ElastomerClient wraps low-level Faraday exceptions into its own error hierarchy so you only need to rescue `ElastomerClient::Client::Error` subclasses in application code.

| Faraday exception | ElastomerClient error |
|---|---|
| `Faraday::TimeoutError` | `ElastomerClient::Client::TimeoutError` |
| `Faraday::ConnectionFailed` | `ElastomerClient::Client::ConnectionFailed` |
| `Faraday::ResourceNotFound` | `ElastomerClient::Client::ResourceNotFound` |
| `Faraday::ParsingError` | `ElastomerClient::Client::ParsingError` |
| `Faraday::SSLError` | `ElastomerClient::Client::SSLError` |
| HTTP 5xx response | `ElastomerClient::Client::ServerError` |
| HTTP 4xx response | `ElastomerClient::Client::RequestError` (or a subclass) |
