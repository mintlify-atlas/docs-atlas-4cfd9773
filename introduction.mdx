ElastomerClient is a Ruby gem that provides a one-to-one mapping of the Elasticsearch HTTP API. It is intentionally low-level: it handles HTTP communication, JSON encoding/decoding, and error classification, while leaving higher-level concerns such as connection pooling, round-robin routing, and retry logic to you and other libraries.

## Design philosophy

ElastomerClient was built when an official Ruby client was not yet available from Elastic. The goals have remained consistent since:

- **One-to-one API mapping.** Every Elasticsearch endpoint is accessible through a corresponding Ruby method. The client avoids adding abstractions that obscure what is being sent to the cluster.
- **Simplicity.** The client does one thing: make HTTP requests to Elasticsearch and return the response. Connection pooling, load balancing, and retry strategies are better handled at different layers.
- **Composability.** Higher-level functionality — bulk indexing, scan/scroll iteration, multi-search — is built from smaller, focused components. This is the UNIX philosophy applied to a Ruby library.

There are a few places where it made sense to wrap the Elasticsearch API inside Ruby idioms. The scan/scroll search type, for example, is exposed as a Ruby iterator rather than requiring you to manage continuation tokens manually.

## Compatibility

<Note>
  ElastomerClient is tested against **Ruby 3.2** and the following Elasticsearch versions: **5.6**, **8.13**, and **8.18**.
</Note>

The gem depends on [Faraday](https://github.com/lostisland/faraday) for HTTP communication, which means you can swap the underlying HTTP adapter at initialization time.
